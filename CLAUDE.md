# CLAUDE.md

Guidance for Claude Code in this repo.

## What this is

Kronwrk: Postgres-backed job scheduler/worker/monitor with CLI admin, in Go. Control-plane service (long-lived `scheduler`/`worker` processes + one-shot commands), not a one-shot CLI. Design rationale/full feature set: [docs/stack_explanation.md](docs/stack_explanation.md); current scope is "walking skeleton" v1. Module path: `kronwrk` (local, not GitHub).

## Commands

```bash
go build -o kronwrk ./cmd/kronwrk
go test ./...                                   # unit tests are DB-free
go test ./internal/schedule/ -run TestNextRun   # single test
go vet ./... && gofmt -l .                      # keep both clean; gofmt -w <file> to fix

./kronwrk migrate                             # run after any new migration file; runs directly (not shell-gated)

# PROJECT POLICY: daemons are started from the interactive shell (`daemon
# start <scheduler|worker>`, admin or operator role) so service_events
# attributes every start/stop to the person's own login. The direct commands
# below still exist (not shell-gated — kept for a future systemd deployment)
# and need creds in DATABASE_URL since TCP requires password auth; if that
# path is ever used, create a least-privilege login via `user add <name>
# --role scheduler|worker` (names "scheduler"/"worker" are reserved). Never
# put the bootstrap superuser (or a full admin) in a daemon's DATABASE_URL.
DATABASE_URL="postgres://<svc-login>:<password>@localhost:5432/kronwrk?sslmode=disable" ./kronwrk scheduler start
DATABASE_URL="postgres://<svc-login>:<password>@localhost:5432/kronwrk?sslmode=disable" ./kronwrk worker start

./kronwrk shell                               # interactive REPL; also opens with no args on a TTY

# everything below only runs inside `kronwrk shell` — direct invocation errors out
job add --name nightly --command /bin/echo --args "hi,there" --schedule "0 2 * * *" --timezone Asia/Kolkata --comment "CHG-1234: why/what"
job list / disable <id> / enable <id>
job condition add <id> --event <name> [--wait 30m]   # event-gate the job's runs (0014): enqueued waiting, promoted once the event arrives; --wait bounds it (omitted = forever), whole seconds; admin only
job condition list <id> / remove <id> --event <name> # removing a job's last condition un-gates its waiting runs (vacuous ALL)
event emit <name> [--payload '{"k":1}']              # satisfies waiting runs on the scheduler's next tick; admin or operator
event list [--limit N]                               # newest first; consumed events dimmed with the run that took them
run status <run-id>
monitor                               # interactive jobs+last-run table (bubbles/table): ↑/↓ move the highlighted row, Enter drills into that job's run history (ListRunsForJob: in-flight runs first via finished_at DESC NULLS FIRST, then newest finished; capped 200; Esc/Backspace returns with the overview cursor intact), F5/r re-query the active view, q quits; piped stdin prints the overview once instead. Any role with SELECT on jobs+job_runs (all but scheduler)
daemon status                         # local process check (pgrep), open to all roles
daemon start <scheduler|worker>       # spawn detached with the session's creds; admin or operator (0013) — support lost daemon access in 0012
daemon stop <scheduler|worker>        # graceful SIGTERM; admin or operator

# user commands need CREATEROLE — run them in the shell as the non-superuser
# user-admin login (a person-named kronwrk_admin+CREATEROLE role; `alice` in
# local dev). The shell REFUSES superuser logins; the bootstrap superuser is
# for `migrate`, psql, and DBeaver only.
user add <name> --role <admin|user|support|scheduler|worker|operator>   # prompts for password
user list / set-role <name> --role <role> / remove <name>
whoami
```

### Local PostgreSQL (Homebrew, keg-only)

Postgres 17's tools aren't on default PATH:
```bash
export PATH="/opt/homebrew/opt/postgresql@17/bin:$PATH"
brew services start postgresql@17
psql -d kronwrk   # unix socket, still trust auth: superuser is the macOS user, no password
```
[`pg.sh`](pg.sh) (repo root) wraps `brew services` for manual start/stop (`./pg.sh start|stop|restart|status`; `start` waits on `pg_isready`). It's a developer convenience, not used by the app — distinct from the shell's own `ensurePostgresRunning` offer-to-start.
`pg_hba.conf` requires `scram-sha-256` for TCP (`host ... 127.0.0.1/32` / `::1/128`); only the unix-socket `local` entry stays `trust`. This is what makes `DATABASE_URL`/shell-login credentials actually enforced instead of silently trusting the OS user — see [pg_hba.conf](/opt/homebrew/var/postgresql@17/pg_hba.conf) (outside the repo; Homebrew-managed).

Default `DATABASE_URL`: `postgres://localhost:5432/kronwrk?sslmode=disable` — has no password, so it will fail to auth over TCP unless credentials are supplied (either embedded in `DATABASE_URL` for daemons/scripts, or via the `kronwrk shell` login prompt for interactive use — see below). Config is env vars only, no config file: `DATABASE_URL`, `POLL_INTERVAL`, `WORKER_CONCURRENCY`, `HEARTBEAT_INTERVAL`, `JOB_TIMEOUT_DEFAULT`, `LOG_LEVEL` (defaults in [internal/config/config.go](internal/config/config.go)).

## Architecture

Three roles coordinate **only** through Postgres — no direct RPC. Scheduler writes rows, worker reads/claims them, DB is source of truth and coordination layer.

- **`cmd/kronwrk/main.go`** — Cobra command tree + process wiring. Commands get config/`db.Store` from the session cache in `session.go` (one shared pgx pool per process, closed in `main` — lets the shell run many commands over one connection). `scheduler`/`worker` install a SIGINT/SIGTERM `signalContext`; one-shot commands use the default context. `interactive.go` = TTY-only wizards (`charmbracelet/huh`); `shell.go` = REPL (`chzyer/readline`), dispatches each line to a **fresh** `rootCmd()` so flag state never leaks between lines; prompts/wizards gated on `stdinIsTTY()` so piped use never blocks. On a TTY, `runShell` runs a fixed startup sequence before the REPL loop starts — covering both `kronwrk shell` and bare `kronwrk`: (1) `printBannerArt` shows the ASCII welcome box; (2) `ensurePostgresRunning` (shell.go) probes the configured host/port with a plain TCP dial (`postgresReachable`, connectivity only — runs before creds are known) and, if nothing answers, offers to `brew services start postgresql@17` and polls up to 10s for it to come up, else aborts with a "start it and re-run" message; (3) `loginPrompt` (in `interactive.go`) asks for a username (**not** prefilled — starts blank; any user embedded in `DATABASE_URL` is used as-is) and a single masked password (`promptExistingPassword` — login proves a known password, so no confirm entry; `user add`'s `promptNewPassword` keeps the double-entry confirm for passwords nobody has typed yet), rewrites the session's `DATABASE_URL` with those creds via `net/url`, and caches it in `sessionCfg` — so the shared pool `getStore` opens on first use connects as the entered user, not silently as the OS user. Then `printConnectionInfo` prints "Connected as …". The REPL loop prints `daemonStatusLine` before **every** prompt (TTY only), so the line above the input area always shows current daemon state — whether the scheduler/worker are up on this machine (green `● up` / faint `○ down`), with a `daemon start|stop` hint only for roles allowed to use it. Daemon control itself lives in `daemon.go`: machine-local by design (like the brew-services offer, **not** cluster orchestration) — `daemon status` pgreps for `kronwrk <svc> start$` (anchored so wrapper shells don't match), `daemon start` respawns this binary detached (`Setpgid`, output to `~/.kronwrk_<svc>.log`) with the **session's own creds** in `DATABASE_URL` and reports early exits with the log tail (2s watch), `daemon stop` SIGTERMs and polls up to 8s for the graceful shutdown; start/stop are admin-or-operator in `cmdPerms` (the `0013` operator role; support lost the service groups in `0012`), systemd deployments keep using the direct `scheduler|worker start` commands. Both also write a `start_requested`/`stop_requested` row to `service_events` over the **shell session's connection** (`logRequested`, same hostname-pid instance id as the daemon's own rows): the daemon's start/stop rows carry the identity it *runs as*, so these are the only record of *who acted* — a SIGTERM has no sender. Best-effort: audit failure warns, never blocks. Steps (3)+pool-open+`refuseSuperuser` are wrapped in `authenticate` (shell.go), shared with `logout`. Piped/scripted `kronwrk shell` skips the banner/Postgres-check/login entirely, using `DATABASE_URL` as configured (and only `refuseSuperuser`, not `authenticate`, runs). The REPL handles three built-ins before Cobra dispatch: `exit`/`quit` (and Ctrl-D) leave; `logout` (TTY-only) tears down the session — `closeSession` + `clearSessionCredentials` (strips the URL's user/pass so the re-login username starts blank) — prints the styled "Logged out" box, then offers log-in-again-or-exit via `confirmWizard`; choosing to continue re-runs `authenticate` and refreshes the readline prompt (`rl.SetPrompt`) + connection banner, otherwise the shell exits. Root's `PersistentPreRunE` gates operator commands (`job`/`run`/`user`/`whoami`) behind the shell's `inShell` flag — direct invocation errors out; `shellExempt` in main.go lists the top-level commands allowed to run standalone (`scheduler`, `worker`, `migrate`, `shell` itself). Add new top-level commands to `shellExempt` explicitly if they must run outside the shell (e.g. for systemd/CI) — the default is shell-only.
- **`internal/db`** — owns the pool ([db.go](internal/db/db.go)), embedded migrations ([migrate.go](internal/db/migrate.go)), and ALL SQL, hand-written on purpose, no ORM/codegen ([queries.go](internal/db/queries.go)) since locking/conflict queries must be explicit. `Store` is the only DB touchpoint for other packages.
- **`internal/scheduler`** — decides WHEN a job runs, never executes it. Each tick: `DueJobs` (`enabled AND next_run_at <= now()`) → `EnqueueRun` inserts a `queued` row + advances `next_run_at`, one transaction — unless the job has `job_conditions` rows (0014, event gating): then the run is born `waiting` with `wait_deadline = scheduled_for + max(wait_seconds)` (NULL when any condition's `wait_seconds` is 0 = wait forever; `waitPlan` in [conditions.go](internal/db/conditions.go)). Two follow-up passes per tick, promotion **before** expiry (an event present at tick time beats a deadline that lapsed in the same interval): `PromoteWaitingRuns` flips waiting→queued when every condition has an unconsumed matching event, consuming them (`consumed_by_run_id`, oldest event first, one run per event) in one tx with `FOR UPDATE SKIP LOCKED` on both run and event rows (invariant-2 discipline, concurrent-scheduler safe); `ExpireWaitingRuns` flips deadline-passed waiting→`skipped` (terminal, `finished_at` set). An event emitted before the scheduled time promotes in the same tick as the enqueue; a job whose conditions were all removed un-gates vacuously. Latency is bounded by `POLL_INTERVAL` (LISTEN/NOTIFY wake-up deferred). Workers are untouched — `ClaimRun` only sees `queued`. On `Run` start/graceful-stop it appends a `start`/`stop` row to `service_events` (`Store.LogServiceEvent`) tagged with a `hostname-pid` instance id plus, since `0009_service_events_identity`, the connected login and its kronwrk group role (`username`/`role_name`, stamped server-side via `current_user` + a WhoAmI-shaped membership lookup — NULL role for logins outside the role model); the stop write uses a fresh 5s context so it lands even though the shutdown ctx is already cancelled.
- **`internal/worker`** — owns execution: fixed-size pool claims `queued` runs, executes, heartbeats the lease, writes terminal status. Logs its own `start`/`stop` to `service_events` the same way as the scheduler (using `w.id` as the instance id).
- **`internal/schedule`** — wraps `robfig/cron` for next-run-time in a job's timezone; only place cron expressions are parsed.

### Correctness invariants (do not break)

1. **Idempotent scheduling.** `job_runs` has `UNIQUE (job_id, scheduled_for)`; `EnqueueRun` uses `ON CONFLICT DO NOTHING`. `scheduled_for` = the job's current `next_run_at` (deterministic, not `now()`) — this is what makes the scheduler restart-safe and overlap-safe. Never weaken the constraint or make `scheduled_for` nondeterministic.
2. **Safe claiming.** `ClaimRun` selects with `FOR UPDATE SKIP LOCKED` and flips to `running` in the same transaction — the only correct way for concurrent workers to poll without double-execution. Never replace with plain `SELECT` + `UPDATE`.
3. **Bounded, killable execution.** Each command runs in its own process group (`Setpgid`); `cmd.Cancel` kills the whole group on timeout; stderr captured in a 1 MiB `boundedBuffer` (stdout discarded until `run_logs` exists). Execution context derives from `context.Background()` (not the shutdown context) so in-flight runs finish on graceful shutdown, bounded by their own timeout. `finalize` uses a fresh short context so terminal status is always written, even during shutdown.
4. **Heartbeats/lease.** A goroutine renews `last_heartbeat_at`/`lease_expires_at` every `HEARTBEAT_INTERVAL`; lease = 3× heartbeat. Exists for future monitor to reclaim dead-worker runs — reclaim logic not yet implemented.

### Access control (RBAC)

Enforced by **Postgres GRANTs, not app logic**. Migration `0003_roles` creates three NOLOGIN group roles (as `supercron_*`; `0004_rename_roles` renames them for the Kronwrk rename): `kronwrk_admin` (full DML, owns tables), `kronwrk_user` (SELECT only), `kronwrk_support` (SELECT + column-level `UPDATE (enabled, updated_at)` on jobs). Migration `0006_service_roles` adds two more NOLOGIN groups for the daemons — least-privilege, **not** admin: `kronwrk_scheduler` (SELECT+UPDATE on jobs, INSERT on job_runs, INSERT on service_events; `0008_scheduler_enqueue_select` adds column-level SELECT on `job_runs (job_id, scheduled_for)` because `INSERT ... ON CONFLICT` needs SELECT on the arbiter-index columns — without it every enqueue 42501s while a plain INSERT preflight passes, which is why `PreflightScheduler`'s test INSERT mirrors the ON CONFLICT clause) and `kronwrk_worker` (SELECT on jobs, SELECT+UPDATE on job_runs, INSERT on service_events). admin owns every table so it already covers every role's access. Migration `0007_support_starts_services` granted `kronwrk_scheduler`+`kronwrk_worker` **to** `kronwrk_support` (so support could run daemons), but `0012_support_no_service_roles` **reverts it** — the escalation proved too confusing. Support is back to exactly its base grants (SELECT everywhere + column `UPDATE (enabled, updated_at)` on jobs) and fails both daemons' preflights. Instead, `0013_operator_role` adds `kronwrk_operator` — the assignable "runs the daemons" role for on-duty staff: **direct** grants (deliberately flat, no group-to-group membership like 0007) covering the union of both daemons' needs plus observability — jobs SELECT+UPDATE, job_runs SELECT+INSERT+UPDATE, service_events SELECT+INSERT. A daemon started by an operator connects as *their* login, so `service_events` attributes start/stop to that person. Every LOGIN role holds exactly one group (enforced by `SetUserRole`'s revoke-all-then-grant transaction over all six). The `GroupRoles` map in [users.go](internal/db/users.go) is the single source of truth for the CLI role names; `ValidateUsername` reserves all six as usernames. User-management SQL in [internal/db/users.go](internal/db/users.go): usernames go through `ValidateUsername` + `pgx.Identifier.Sanitize()`; passwords are hashed client-side to a SCRAM-SHA-256 verifier (`buildSCRAMVerifier`, RFC 8265 OpaqueString prep matching pgx) before being embedded via `quoteLiteral` — DDL can't bind params, and a plaintext `PASSWORD '...'` would land in server logs under `log_statement='ddl'`. Keep both. CLI's only enforcement role is presentation: `permissionErr` in main.go maps pg error 42501 to a friendly message; `PreflightScheduler`/`PreflightWorker` fail daemons fast at startup (loops otherwise log-and-continue); and the shell is role-aware via `cmdPerms` in [permissions.go](cmd/kronwrk/permissions.go) — a client-side mirror of the GRANT matrix that renders inaccessible commands faint in shell `help` (annotated with what they require) and refuses them pre-dispatch (`authorizeShellCmd`, called from root's `PersistentPreRunE`) with a "not authorized" message instead of a post-round-trip 42501. `user add/remove/set-role` key on the `rolcreaterole` attribute (surfaced as `DBUser.CreateRole` via `WhoAmI`), so a plain admin sees them dimmed while the user-admin login doesn't. Presentation only — Postgres still enforces if the map drifts — but keep `cmdPerms` in sync when a migration changes grants; `TestCmdPermsMatchCommandTree` forces every shell leaf command to have an entry. The connected user is cached per session (`currentUser` in session.go, cleared by `closeSession`/logout). Migration `0014_event_conditions` adds the event-gating tables and grants: `events`/`job_conditions` owned by admin, SELECT to user/support/operator; `INSERT, UPDATE ON events` to operator (emit + consume when running the scheduler daemon); scheduler gets SELECT on job_conditions, SELECT+UPDATE on events, **table-level** UPDATE on job_runs (its promotion `SELECT ... FOR UPDATE` needs it — same table-level-for-row-locks lesson as 0006) while its job_runs SELECT stays column-scoped (`id, status, wait_deadline` on top of 0008's `job_id, scheduled_for`), so it still can't read run results — which keeps `event list` (full-row SELECT) off the scheduler role in `cmdPerms` while `job condition list` includes it.

**Shell refuses superuser.** `refuseSuperuser` (shell.go, via `WhoAmI`'s `rolsuper`) blocks the interactive shell from operating as a Postgres superuser — a superuser bypasses every GRANT, and over a trust socket the login password isn't even verified. Superuser is confined to `migrate`, psql, and DBeaver. `user add/remove/set-role` need CREATEROLE (role attributes aren't inherited via group membership), so user management runs as a **non-superuser user-admin login** (`alice` in local dev; called the *user-admin* login — distinct from the `operator` group role): a full `kronwrk_admin` (inherits DML on jobs/job_runs) that *also* has CREATEROLE and ADMIN OPTION on all six groups, so it can create users and assign any of the six roles — all without being a superuser. It must be **person-named, not `admin`** — `ValidateUsername` reserves the role names (admin/user/support/scheduler/worker/operator) so no login collides with a role. Bootstrap it once as the superuser, outside the shell:
```sql
CREATE ROLE alice LOGIN PASSWORD '<pw>' NOSUPERUSER NOCREATEDB CREATEROLE INHERIT;
GRANT kronwrk_admin TO alice WITH ADMIN OPTION;                                   -- inherited: data DML + grantable
GRANT kronwrk_user, kronwrk_support, kronwrk_scheduler, kronwrk_worker, kronwrk_operator
    TO alice WITH ADMIN OPTION, INHERIT FALSE, SET FALSE;                         -- admin-only, for user add/set-role's grant+revoke
```
`WhoAmI`/`ListUsers` filter on `am.inherit_option`, so the user-admin login's admin-only non-admin grants don't surface it as a multi-role user — it displays as plain `admin` (its only inherited group). Migrations `0006`/`0013` create the service/operator groups but can't grant them to a per-deployment login, so the `GRANT ... TO alice` above is the bootstrap step that lets `user add --role scheduler|worker|operator` work.

The daemons run under their own narrow groups, not admin: `kronwrk_scheduler`/`kronwrk_worker` (migration `0006_service_roles`) grant exactly what `PreflightScheduler`/`PreflightWorker` and the loops need — the preflights' `SELECT ... FOR UPDATE` require **table-level** UPDATE (jobs for scheduler, job_runs for worker), which is why those grants aren't column-scoped. Because table-level UPDATE on jobs would otherwise let the scheduler/operator roles rewrite `command`/`args` (a worker then executes it = privilege escalation to RCE), migration `0015_lock_job_command` adds a BEFORE UPDATE trigger (`kronwrk_forbid_job_command_rewrite`) that rejects changes to `jobs.command`/`jobs.args` from any login that is not a member of `kronwrk_admin`. No app path edits those columns (jobs are immutable after `job add`), so this is transparent to `next_run_at`/`enabled` updates and the `FOR UPDATE` locking; if a `job edit` feature is ever added, it must run as admin. Create a role-scoped login via `user add <name> --role scheduler|worker` (not the old admin login, and not the superuser) and put its creds in the daemon's `DATABASE_URL`. `admin` logins can also run either daemon (table owner), as can `operator` (its whole purpose — the on-duty role, attributed by login in service_events), so the preflight passes for scheduler/operator/admin (scheduler start) and worker/operator/admin (worker start); support cannot since `0012`.

### Migrations

`.sql` files in `internal/db/migrations/`, embedded via `//go:embed` in [migrate.go](internal/db/migrate.go) — must stay co-located with it. golang-migrate used as a library; `RunMigrations` rewrites `postgres://` to the `pgx5://` scheme. Add new migrations as paired `NNNN_name.up.sql`/`.down.sql`; apply with `./kronwrk migrate`.

RBAC conventions since `0003_roles`:
- Any migration creating a table must add explicit ownership/grants: `ALTER TABLE <t> OWNER TO kronwrk_admin;` + `GRANT SELECT ... TO kronwrk_user, kronwrk_support;` per the matrix. (0003's `ALTER DEFAULT PRIVILEGES` only covers tables created by the role that ran 0003.)
- Down migrations touching roles must `OWNER TO CURRENT_USER` before any `DROP OWNED BY` — wrong order drops the tables. See `0003_roles.down.sql`.

### Deferred (schema columns exist, logic does not)

v1 omits: monitor/stale-lease reclaim, retry handling (`max_retries`/`attempt`), overlap policy (`allow_overlap`/`max_concurrent_runs`), misfire policy (`misfire_policy`), `LISTEN/NOTIFY`, `notifications`/`workers`/`run_logs` tables. Columns already exist in `0001_init` so these slot in without migration churn — keep the coordinate-through-Postgres model and the four invariants above.

Event gating (0014) ships without: a file-watcher daemon (events are emitted via `event emit` or SQL INSERT by a scoped login — a future `kronwrk watcher` would translate fsnotify into event rows), NOTIFY-based low-latency promotion (poll-tick only; if added, NOTIFY must stay a wake-up signal — the events table is the durable source of truth), event GC/pruning (unconsumed events live forever and match with no lower time bound), and re-running a `skipped` occurrence (permanent under invariant 1; would need a `job trigger` feature).
